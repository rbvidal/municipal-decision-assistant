## What to Build

A complete enterprise case-management application for German municipal employees. The platform helps Sachbearbeiter process administrative cases — building permits, procurement requests, personnel actions, citizen inquiries — from inbox to archive. Decision support tools quietly assist in the background.

This is **Microsoft Outlook meets Jira meets SAP Fiori** — a professional desktop work tool, not a consumer app, not an AI chatbot.

The backend is a Spring Boot REST API with JWT authentication. All endpoints return JSON. The frontend generated by Stitch completely replaces the existing Thymeleaf UI.

## Target Users

**Primary:** Sachbearbeiter in German municipalities. Domain experts in public administration. Process 10-30 cases per day. Work on desktop workstations (1280-1920px).

**Secondary:** Supervisors who review and approve. Department administrators who manage documents. System administrators who monitor infrastructure.

## Application Name

**Kommunale Entscheidungsplattform** (Municipal Decision Platform)

## Language

**German.** Every label, button, error message, notification, and help text must be in German. DE/EN toggle in user menu is the only exception.

## Primary Screens

### 1. Startseite (Home — Operational Dashboard)

**Design rule: Work first. No AI prompt on this screen.**

The user lands here after login and sees their workload:

```
Six stat cards: Meine Vorgänge (12) | Heute fällig (3) | Überfällig (2) |
                Wartet auf Bürger (4) | Wartet auf Behörde (1) | Heute erledigt (5)

Two-column layout (70/30):
  LEFT (70%): Case lists
    - Überfällig (red highlight, days overdue)
    - Heute fällig (time + priority)
    - Wartet auf Bürger (days waiting, reminder action)
    - Wartet auf andere Behörde (days waiting, follow-up action)
    - Heute erledigt (completed today with result + who)
    - Von mir beobachtet (watched cases, not assigned to me)
    - Vorgeschlagene nächste Aufgabe (one recommended case with risk indicator)

  RIGHT (30%): Decision Support sidebar
    - Compact question input: "Frage zu Vorschriften oder Verfahren?"
    - Context-aware suggestions based on active cases
    - Not the centerpiece. A reference tool.
```

### 2. Meine Arbeit (My Work — Case Workspace)

The heart of the product. Sub-navigation:

```
[ Posteingang (3) ]  [ Offene Vorgänge (12) ]  [ Warten (5) ]  [ Genehmigung (2) ]  [ Archiv (47) ]
```

**Case lifecycle:** Posteingang → Prüfung → Entscheidungsunterstützung → Entwurf → Genehmigung → Versand → Archiv

At any point, a case can enter: Wartet auf Bürger | Wartet auf Behörde | Wartet auf Kollegen

**Case detail (three-panel layout):**
- **Left (240px):** Case metadata, assignment, priority, deadline, risk indicator
- **Center (flex):** Workspace — phase indicator, dynamic checklist, documents, internal notes, activity timeline
- **Right (320px):** Decision Support panel — summary, applicable regulations, missing information, suggested checklist items, suggested next action, supporting documents. "▶ Erweitert" toggle hides all technical details.

**Key features visible in every case:**
- **Checklist** — dynamic, adapts to case type. Items auto-populate. User can add/remove.
- **Internal notes** — author + timestamp. Never sent to citizens. Searchable.
- **Activity timeline** — who did what, when. Full audit trail behind "Vollständigen Verlauf".
- **Risk indicator** — 🟢 Gering / 🟡 Mittel / 🔴 Hoch. Computed from deadline proximity, complexity, missing docs, waiting duration.
- **Missing documents** — prominent warning if required documents are absent. "Vom Bürger anfordern" button.
- **Assignment** — shows current assignee, reassign with reason dropdown.

**Supervisor approval view:**
- Side-by-side: original application vs. draft decision
- Auto-detected changes highlighted
- Auto-check: "Alle zitierten Vorschriften sind aktuell", "Rechtsbehelfsbelehrung korrekt", "Gebühren korrekt", "Vier-Augen-Prinzip", "Fristen eingehalten"
- Approve / Return for revision / Reject

### 3. Wissen (Knowledge — Unified Search)

Search across **everything.** Results grouped by category with dividers:

```
Sub-tabs: [ Alles ] Vorschriften Verfahren Vorlagen FAQs Fälle Bürger Dokumente

Search results grouped:
  ── VORSCHRIFTEN (3) ──    (relevance bar, excerpt, section ref, doc link)
  ── VERFAHREN (1) ──       (procedure title, steps, department)
  ── FAQs (2) ──            (question, department, answer preview)
  ── VORLAGEN (1) ──        (template name, fields, usage count)
  ── FÄLLE (4) ──           (case #, subject, status, citizen, link)
  ── BÜRGER (1) ──          (name, address, case count, citizen file link)
  ── DOKUMENTE (1) ──       (title, type, pages, department)
```

Archived cases and internal notes are searchable. Each category shows top 5 results with "Mehr anzeigen" link.

### 4. Dokumente (Documents)

Sub-tabs: Alle Dokumente | Hochladen | Index-Status

- **Document list:** Dense table (50 rows). Row actions: view, metadata, new version, version comparison, related documents, referenced regulations.
- **Document detail:** Metadata, related documents, version history with comparison, referenced/citing regulations, full text. "▶ Erweitert: Index-Details" for technical info (ADMIN only).
- **Version comparison:** Side-by-side diff. Auto-detected change summary. Export as PDF.
- **Upload:** Drag-and-drop zone, metadata form, progress indication.

### 5. Verwaltung (Administration)

ADMIN role only. Tool grid landing page linking to: Korpus-Status, Audit, Aufträge, Benchmarks, Entwickler, Systemkonfiguration. All developer and infrastructure screens live here — never in the main navigation.

### 6. Authentication & Errors

Full login/forgot/reset/first-login/session-expired/logout flows with German labels. Error screens for 401, 403, 404, 500, backend unavailable, decision support unavailable, Qdrant unavailable.

## Visual Style

**Enterprise desktop software. German public administration. High information density.**

- Color: Blue-gray (#2B6CB0 primary). Semantic green/amber/red. White cards, gray-50 background. NO shadows, NO gradients, NO glassmorphism.
- Typography: Inter (body), JetBrains Mono (IDs). 14px body, 13px tables, 24px headings.
- Tables: 40px row height, 50 rows default, sticky headers, alternating rows, hover highlight. **Tables are the primary UI pattern.**
- Cards: White, 1px gray-300 border, 8px radius. No shadows.
- Buttons: 36px height. Primary (blue-700), secondary (white+border), danger (red-700), ghost.
- Layout: Max 1400px content width. 32px horizontal padding. 3-5 column card grids.
- Navigation: 56px top bar, 44px sub-nav, 28px breadcrumb.
- Information density: HIGH. Think Jira issue view, not marketing landing page.
- Language: All German. Professional municipal tone. Never use "magic", "smart", "AI", "KI" in labels.
- Decision Support (not "AI"): The system supports decisions. It says "Zusammenfassung" not "KI-Zusammenfassung". Technical metrics hidden behind "▶ Erweitert".

**DO NOT use:** Gradients, glassmorphism, shadows, oversized cards, purple AI aesthetics, sparkle icons, "magic" language, hero layouts, mobile-first patterns.

## User Personas

1. **Sachbearbeiter:** Case worker. Opens inbox, processes applications, checks regulations, drafts decisions. Prioritizes by deadline and risk. Takes internal notes. Sends to supervisor.
2. **Supervisor:** Approves/rejects/revisions. Compares application to draft. Sees audit trail. Monitors team workload.
3. **Department Administrator:** Uploads/maintains document collection. Checks indexing status. Manages document lifecycle.
4. **System Administrator:** Monitors infrastructure. Investigates failures. Runs benchmarks. Trace documents through pipeline.

## Key Flows

1. **Daily work:** Login → Startseite (see 12 open, 2 overdue, 3 due today) → Click suggested next task → Review documents → Read decision support → Complete checklist items → Add internal notes → Draft decision → Submit for approval

2. **Case goes to waiting:** Open case → Realize document missing → Set "Wartet auf Bürger" → System tracks days waiting → Citizen submits document → Case returns to active → Continue processing

3. **Supervisor approval:** Login → Genehmigung tab (2 pending) → Open case → Side-by-side review → Check auto-verification points → Add approval comment → Genehmigen

4. **Research:** Open Wissen → Search "Abstandsflächen Carport" → See results grouped: Vorschriften (3), FAQs (2), archived Fälle (2), Vorlagen (1) → Click regulation → Copy citation → Return to case

5. **Document comparison:** Dokumente → Select document → Versionen → v1 ↔ v2 vergleichen → Side-by-side diff → See auto-detected changes → Export

## API Endpoints (UI-Relevant Only)

**Auth:** POST /api/auth/login, /register, /refresh, /logout; GET /api/auth/me
**Decision Support:** POST /decision (question → answer, sources, trace)
**Search:** POST /api/search (query, mode → results with scores)
**Documents:** GET /api/documents, GET /api/documents/{id}, GET /api/documents/{id}/content, PATCH /api/documents/{id}/metadata, POST /api/documents/{id}/archive, DELETE /api/documents/{id}, POST /api/documents/{id}/reindex, DELETE /api/documents/{id}/purge
**Upload:** POST /documents/upload (multipart), POST /documents/batch
**Ingestion:** POST /api/document-ingestion-jobs/documents/{id}, GET /api/document-ingestion-jobs
**Workspaces:** GET/POST /api/workspaces, GET /api/workspaces/{id}, POST /api/workspaces/{id}/advance, GET /api/workspaces/{id}/documents, GET /api/workspaces/{id}/timeline
**Audit:** GET /api/audit/events
**Corpus:** POST /admin/corpus-inventory/generate, POST /admin/corpus-release-report/generate
**Providers:** GET /api/providers/status, GET /api/providers/models
**Dev:** GET /dev/knowledge/salary, /travel, /thresholds; GET /dev/perf

Three endpoints need JSON wrappers (P1): POST /api/decision, GET /api/corpus/health, GET /api/corpus/inventory. Backend logic exists — only response format changes.
