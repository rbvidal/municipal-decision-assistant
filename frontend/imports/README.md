# Stitch Imports — Immutable Design Source

## Purpose

This directory stores **original, unmodified Google AI Studio (Stitch) exports.** Each export is a complete Vite + React project generated by Stitch for one screen. These files are the design source of truth — they are never edited.

## Immutability Rule

**Nothing inside `imports/` is ever edited manually.** These files are a permanent, versioned archive of what Stitch generated. If a design must change, a new Stitch export is created as a new version. Previous versions are preserved for comparison.

## Workflow

### 1. Generate

Generate a screen in Google Stitch using the UX specification (`docs/STITCH_SPECIFICATION.md`) and design system (`docs/DESIGN.md`).

### 2. Export

Export the generated screen from Google AI Studio as a downloadable Vite project.

### 3. Import

Copy the **entire exported project unchanged** into:

```
imports/<screen>/v1/
```

If this is a redesign of an existing screen, create a new version:

```
imports/<screen>/v2/
imports/<screen>/v3/
```

### 4. Analyse

DeepSeek analyses the imported export. It identifies:

- Reusable components (tables, buttons, cards, navigation)
- Page-specific layout patterns
- CSS that maps to the design system tokens
- API call patterns that match the backend endpoints
- Differences from previous versions (if v2, v3, etc.)

### 5. Extract

Reusable components are extracted from the import and adapted into:

```
src/components/<category>/<ComponentName>.tsx
```

Adaptations may include: TypeScript typing, prop interfaces, design token mapping, API client integration, and accessibility enhancements.

### 6. Merge

Page-level code is merged into:

```
src/pages/<screen>/
```

The merge adapts the Stitch-generated code to the production codebase conventions while preserving visual fidelity.

### 7. Document

Every merge is logged in `docs/MERGE_LOG.md` with the source version, components extracted, and decisions made.

## Versioning

- `v1` — Initial Stitch export for a screen
- `v2` — First redesign iteration
- `v3` — Second redesign iteration
- etc.

Versions are never deleted. They form a permanent design history.

## Relationship to stitch-export/

The project root `stitch-export/` directory stores screenshots and design reference files extracted from the same Stitch exports. While `stitch-export/` is a visual reference for stakeholders, `imports/` is the technical source for component extraction.

## Constraints

- Never run `npm install` inside an import directory.
- Never edit any file inside an import directory.
- Never commit `node_modules/` from an import (they are gitignored).
- Never merge code directly from an import into `src/` without review and adaptation.
